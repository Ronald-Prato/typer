Typer - Plan de cierre de escalabilidad

Este plan convierte la auditoria actual en un checklist ejecutable. El objetivo no es solo pasar lint/tests, sino dejar evidencia suficiente para que una siguiente auditoria pueda revisar backend, frontend, datos, concurrencia y pruebas sin encontrar huecos obvios de arquitectura.

Veredicto de partida

Base sanaRefactor anterior

El repo ya tiene reglas puras en src/domain, helpers en convex/*State.ts y lifecycle en convex/*Lifecycle.ts. Los gates no lanzadores pasan.

EjecutadoEscala real

Se cerraron los puntos de query shape, paginacion, matchmaking stale-safe, bot snapshot y componentes principales con separacion controller/vista.

CriterioAuditoria futura

Una auditoria favorable debe poder encontrar indices para busquedas, cursores para listas crecientes, pruebas de carreras y componentes/hook boundaries claros.

Checklist de cierre

Area Problema actual Estado objetivo Evidencia requerida
Busqueda social searchUsersByNickname filtra usuarios sin indice por nickname normalizado. Campo nicknameSearch normalizado e indexado. Busqueda por prefijo usando withIndex. Schema actualizado, helper puro testeado, integration tests de busqueda, limite de resultados y no auto-inclusion.
Listas sociales Amigos y solicitudes usan lecturas completas o no limitadas. APIs limitadas con indices por fecha para amigos/solicitudes, con compatibilidad razonable para la UI actual. Tests de limites, empty states y orden estable. UI usando limites/cursor donde aplique.
Historial La UI usa limit/page, que crece linealmente en paginas profundas. UI usando paginationOpts con cursor real de Convex. API legacy de offset eliminada. Tests de normalizeHistoryPageSize y componente/hook o codigo inspeccionable sin page.
Matchmaking La mutation mezcla seleccion, pairing, creacion de juegos, scheduling de bot y patches. Seleccion y guards puros en queueState. Mutations con revalidacion justo antes de escribir o agendar bot. Tests para pares, impar, usuarios ya movidos, lote maximo e idempotencia parcial.
Bot Un usuario global imabot se muta para todas las partidas. Identidad visual de bot congelada por juego o derivada de forma deterministica, sin mutar un perfil global por match. Schema/DTO actualizado, tests que prueben matches simultaneos sin pisar avatar/nickname global.
Lifecycle de juego Finalizar juego limpia solo al usuario actual; hay riesgo de estados colgados si el flujo no completa. Limpieza explicita de humanos afectados en reject/finish, conservando historial y winner server-authoritative. Integration tests de finalizar/ganar/rechazar y repeated calls.
Frontend modal/lifecycle Modales usan polling de setInterval para detectar cierre. Modal expone callbacks de cierre; consumidores no hacen polling. Tests de modal/cierre o inspeccion sin intervalos en modales de perfil/amigos.
Frontend 1v1 MatchMakingComponent y MatchProgress siguen grandes y mezclan comandos, shortcuts y vista. Hooks de orquestacion y sub-vistas presentacionales. Reglas compartidas para shortcuts/progreso. Archivos por debajo del umbral o con responsabilidad unica, y tests para reglas extraidas.

Fase 1 - Backend server-authoritative

  1. Crear helpers puros para busqueda social: normalizeNicknameSearch y tamano defensivo de pagina social.
  2. Agregar nicknameSearch al schema y a create/update, indexado para prefijo.
  3. Cambiar searchUsersByNickname a withIndex y limitar resultados.
  4. Acotar/paginar amigos y solicitudes. Si la UI aun necesita una lista simple, mantener un limite defensivo documentado.
  5. Migrar historial UI a cursor pagination real.

Fase 2 - Matchmaking y bot resilientes

  1. Extraer pairing puro desde queue.ts: seleccion, pares humanos, usuario impar y max batch.
  2. Revalidar cada usuario justo antes de crear/patchar una partida.
  3. Eliminar mutacion de perfil global del bot; guardar botOpponent o snapshot equivalente en el documento de juego.
  4. Agregar tests de lote, impar, usuario ya movido, bot snapshot y reintentos.

Fase 3 - Frontend mantenible

  1. Cambiar polling de modales por callback/ref lifecycle.
  2. Extraer hooks de matchmaking para aceptar/rechazar/salir y shortcuts.
  3. Extraer reglas puras de progreso 1v1: conteo de steps, etiquetas y ganador.
  4. Reducir componentes grandes solo donde el split mejore responsabilidad real; no hacer cortes mecanicos.

Fase 4 - Documentacion y gates

  1. Actualizar AGENTS.md con reglas concretas de indices, paginacion, bot snapshots, matchmaking y no-polling.
  2. Ejecutar tests focalizados mientras se implementa.
  3. Cerrar con pnpm typecheck, pnpm test y pnpm lint, sin lanzar la app.
  4. Hacer auditoria final contra esta tabla, no contra memoria o intencion.
pnpm typecheck pnpm test pnpm lint

Regla para futuras auditorias

Si aparece una nueva lista que puede crecer por usuario, debe tener limite, cursor o indice antes de considerarse lista para produccion. Si aparece una mutation que cambia matchmaking, rewards, history o permisos, debe existir una regla pura testeada que describa la transicion. Si aparece un bot, su identidad de partida debe ser inmutable por juego.
